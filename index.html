Root Cause Chains
Each chain starts from the observable failure and asks "why" until it hits a system design decision. No solutions — just diagnosis.
Pain Point 1A — Expectation Mismatch: "Thoda jaldi nahi ho sakta kya?"
!
Observable Failure: ~90 customers (~16%) cancelled within 12h with zero system failures. They simply expected install to happen faster.
↓
↓
Why? Mental model is same-day activation, benchmarked against Jio, Airtel and local cable.
↓
↓
Why? Booking journey did not anchor expected install TAT upfront. No explicit timeline acknowledgment before booking.
↓
•
Pre-booking expectation setting on installation speed is weak. System takes booking without forcing customer to confirm they understand the install TAT.
System Design Flaw

!

Pain Point 1C — Accumulated System Failure: "Bohot der se wait kar rahe hai"
!
Observable Failure: 127 customers (22%) cancelled after 2+ failures. 9 experienced 5+ failures, avg 258h wait. After each failure, routed back into same loop.
↓
↓
Why? Repeated failed attempts exhausted trust. Customer re-entered the same scheduling loop each time.
↓
↓
Why? Failed cases not differentiated from first-time installs. No escalation-based process exists.
↓
•
Repeated-failure customers are routed back into BAU instead of a recovery workflow. System has no notion of a "failed case" that needs different handling.
System Design Flaw

!

↓
↓
Why? Customers could not understand booking status. No clear, reliable update available anywhere.

Why? CC, bot and tech don't reflect the same information. Partner-side updates not synced to customer-facing systems in real time.

No single source of truth for customer's install state. Every channel sees a partial view, so customer cannot get a reliable answer from anyone.

System Design Flaw
Repeated-failure customers are routed back into BAU instead of a recovery workflow. The system has no notion of a "failed case." Each failure adds ~30h of wait and another CX call cycle.
